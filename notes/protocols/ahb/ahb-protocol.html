AMBA AHB Protocol

📅 Add date ⏱ 12 min read 🏷 HREADY · HTRANS · Arbitration · Pipelined

What is AHB?

AHB (Advanced High-performance Bus) is part of the ARM AMBA bus family, designed for moderate-performance on-chip communication. Unlike AXI4 which has independent point-to-point channels, AHB is a shared bus — all masters and slaves share the same address and data lines, with an arbiter controlling who gets bus access at any time.

💡 Key difference from AXI: AHB has one address bus and one data bus shared by all components. AXI4 gives each master-slave pair independent channels. AHB is simpler and lower power; AXI4 is higher throughput.

HTRANS — Transfer Types

The master uses HTRANS to tell the bus what kind of transfer is happening each cycle.

ValueNameMeaning
2'b00IDLENo transfer requested. Master is not using the bus.
2'b01BUSYMaster owns the bus but needs a gap mid-burst (not ready for next beat yet).
2'b10NONSEQFirst beat of a burst, OR a single (non-burst) transfer.
2'b11SEQSubsequent beats of a burst — address is sequential from previous.

✅ Memory tip: Think of it as — IDLE = doing nothing, NONSEQ = starting something new, SEQ = continuing what was started, BUSY = pause mid-burst.

Pipelined Transfer — Address & Data Phases

AHB is pipelined: the address phase of transfer N+1 overlaps with the data phase of transfer N. This means the address and data of the same transfer appear on the bus in different cycles. 

          | Cycle 1        | Cycle 2        | Cycle 3
----------+----------------+----------------+----------------
HADDR     | 0x1000 (T1)    | 0x1004 (T2)    | 0x1008 (T3)
HTRANS    | NONSEQ         | SEQ            | SEQ
HWDATA    | --- (no data)  | 0xDEAD (T1)    | 0xBEEF (T2)
HREADY    | 1              | 1              | 1

⚠️ Monitor trap: Because of pipelining, you cannot capture address and data in the same cycle. Your monitor must register the address in cycle N and capture the data in cycle N+1 (when HREADY=1).

Wait States — HREADY

A slave inserts wait states by holding HREADY = 0. The master must hold its address and control signals stable until HREADY = 1. 

          | Cycle 1      | Cycle 2      | Cycle 3      | Cycle 4
----------+--------------+--------------+--------------+--------------
HADDR     | 0x1000       | 0x1000 held  | 0x1000 held  | 0x1004
HTRANS    | NONSEQ       | NONSEQ held  | NONSEQ held  | SEQ
HREADY    | 0 (wait)     | 0 (wait)     | 1 (accept)   | 1
          |        2 wait states inserted by slave

HRESP — Slave Response

ValueNameMeaning
2'b00OKAYTransfer successful
2'b01ERRORTransfer failed — slave must drive this for 2 cycles

⚠️ ERROR response is 2 cycles: When a slave signals ERROR, it drives HREADY=0 in the first cycle and HREADY=1 in the second. This gives the master one extra cycle to see the error and optionally abort the burst.

Full Write Transaction — Step by Step

Cycle 1 -- ADDRESS PHASE
  HADDR  = 0x4000_0000   (target address)
  HTRANS = NONSEQ        (start of new transfer)
  HWRITE = 1             (write direction)
  HSIZE  = 3'b010        (word = 32-bit)
  HBURST = SINGLE
  HSEL   = 1             (decoder selected this slave)
  HREADY = 1             (slave ready, no wait states)

Cycle 2 -- DATA PHASE
  HWDATA = 0xDEAD_BEEF   (write data now driven)
  HTRANS = IDLE          (no next transfer)
  HREADY = 1             (slave accepted the data)
  HRESP  = OKAY

Multi-Master Arbitration

When multiple masters want the bus simultaneously, an arbiter grants access to one at a time using request and grant signals.

Per master:
  HBUSREQx  -- Master x requests bus ownership
  HLOCKx    -- Master x wants a locked (atomic) sequence
  HGRANTx   -- Arbiter grants bus to master x

Master gains bus ownership when:
  HGRANTx == 1  AND  HREADY == 1  on posedge HCLK

💡 Why no arbiter in AXI4? AXI4 uses a crossbar interconnect — each master has its own dedicated channels, so multiple masters transfer simultaneously without competing for a shared bus.

Verification Checklist

  • Verify HWDATA is stable exactly one cycle after the address phase (pipelining check)
  • Test all HTRANS transitions: IDLE→NONSEQ, NONSEQ→SEQ, SEQ→BUSY→SEQ, SEQ→IDLE
  • Test wait state insertion: HREADY=0 for 1, 2, and many cycles; verify master holds signals stable
  • Test ERROR response: verify slave drives 2-cycle error and master correctly sees HRESP=ERROR
  • Test all burst types: SINGLE, INCR, INCR4, WRAP4, INCR8, WRAP8, INCR16, WRAP16
  • Test back-to-back transfers with no IDLE gap between them
  • For multi-master: verify arbitration priority, verify HLOCK prevents preemption mid-burst
  • Verify HSEL is asserted only for the correct slave address range (decoder logic)
