AMBA AXI Protocol

📅 Add date ⏱ 15 min read 🏷 VALID/READY · 5-Channel · Burst · SLVERR

What is AXI?

AXI (Advanced eXtensible Interface) is part of the ARM AMBA bus family. AXI4 is the high-performance variant — designed for high-bandwidth, high-frequency on-chip communication between masters (CPU, DMA) and slaves (memory, peripherals). Key advantage over AHB: independent channels allow parallel read and write, and out-of-order responses via ID tags.

💡 Core idea: AXI separates address and data into independent channels with their own handshake, allowing pipelined and out-of-order transactions — unlike AHB's shared bus.

The 5 Channels

Every AXI transfer uses some combination of these 5 independent channels. Each has its own VALID/READY handshake.

ChannelKey SignalsDirectionPurpose
AW — Write AddressAWADDR, AWLEN, AWSIZE, AWBURST, AWIDMaster → SlaveSend write address + burst info
W — Write DataWDATA, WSTRB, WLASTMaster → SlaveSend write data beats
B — Write ResponseBRESP, BIDSlave → MasterConfirm write done / report error
AR — Read AddressARADDR, ARLEN, ARSIZE, ARBURST, ARIDMaster → SlaveSend read address + burst info
R — Read DataRDATA, RRESP, RLAST, RIDSlave → MasterReturn read data beats

VALID / READY Handshake

A transfer completes on a channel only when both VALID and READY are HIGH on the same rising clock edge.

Rule 1: Sender drives VALID when its data is ready to transfer
Rule 2: Receiver drives READY when it can accept
Rule 3: Transfer = VALID & READY == 1 on posedge clk
Rule 4: Master CANNOT deassert VALID once asserted (until handshake done)
Rule 5: Slave CAN deassert READY at any time (back-pressure)

⚠️ Deadlock trap: Never make VALID dependent on READY. If master waits for READY before asserting VALID, and slave waits for VALID before asserting READY — neither fires. VALID must be driven independently.

Burst Types (AWBURST / ARBURST)

ValueNameAddress per beatUse case
2'b00FIXEDSame address every beatFIFO / streaming to one register
2'b01INCRIncrements by transfer sizeNormal memory read/write
2'b10WRAPIncrements, wraps at boundaryCache line fills

Burst length = ARLEN + 1. So ARLEN=0 → 1 beat; ARLEN=7 → 8 beats.

Response Codes (BRESP / RRESP)

CodeNameMeaning
2'b00OKAYNormal successful transfer
2'b01EXOKAYExclusive access successful
2'b10SLVERRSlave error — transfer failed at slave
2'b11DECERRDecode error — address not mapped to any slave

AXI4 vs AXI4-Lite vs AXI4-Stream

FeatureAXI4AXI4-LiteAXI4-Stream
Burst support✅ Up to 256 beats❌ Single transfer onlyN/A (continuous)
Out-of-order✅ Via ID tags❌❌
Address channels✅ AW + AR✅ AW + AR❌ No address
Typical useDDR / high-perf memoryRegister maps (CSRs)Video / data pipeline

✅ Rule of thumb: Use AXI4-Lite for register access (like camera CSRs at Qualcomm). Use full AXI4 for DDR or DMA transfers. Use AXI4-Stream for pixel pipelines.

Write Transaction — Cycle by Cycle

Cycle 1 | AW: AWADDR=0x1000, AWLEN=1, AWBURST=INCR, AWVALID=1
        |      Slave: AWREADY=1  →  Address handshake done

Cycle 2 | W:  WDATA=0xDEAD, WSTRB=0xFF, WLAST=0, WVALID=1
        |      Slave: WREADY=1   →  Beat 1 done

Cycle 3 | W:  WDATA=0xBEEF, WLAST=1, WVALID=1
        |      Slave: WREADY=1   →  Beat 2 done (last)

Cycle 4 | B:  Slave drives BRESP=OKAY, BVALID=1
        |      Master: BREADY=1  →  Response handshake done

Outstanding Transactions & IDs

AXI4 allows multiple in-flight transactions using ID tags. Responses must return in-order per ID, but can be out-of-order across different IDs.

// Two reads in-flight simultaneously
Cycle 1: AR — ARID=0, ARADDR=0x1000
Cycle 2: AR — ARID=1, ARADDR=0x2000

// RID=1 can return before RID=0 — valid in AXI4!
// Within the same ID, order is preserved.

Verification Checklist

  • Cover all 3 burst types (FIXED, INCR, WRAP) in covergroups
  • Test SLVERR by accessing an unmapped address region
  • Test back-pressure: slave holds WREADY / RREADY low for N cycles
  • Test narrow transfers: WSTRB with only some byte lanes active
  • Verify WLAST aligns with AWLEN — mismatch is a common RTL bug
  • Test maximum burst length (ARLEN=255) and minimum (ARLEN=0)
  • Test out-of-order responses with multiple simultaneous IDs
